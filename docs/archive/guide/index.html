01 What is LobsterFarm?

LobsterFarm turns a single machine into a software team. A local daemon orchestrates specialized AI agents -- each with a defined role, memory, and personality -- that plan features, write code, review pull requests, and ship software with minimal human intervention.

The system runs as a Node.js daemon managed by launchd, with agents communicating through Discord channels and collaborating through Git. It is not a wrapper around a single AI prompt. It is an orchestration platform where six agent archetypes form a pipeline: features enter as ideas, move through planning, design, implementation, code review, and ship as merged pull requests.

What makes it different

Specialization over generalism. Each agent loads different DNA (system instructions, domain knowledge, coding standards) depending on the task. A Planner agent runs socratic discovery sessions to write specs. A Builder agent implements code in isolated worktrees. A Reviewer agent audits PRs with fresh eyes. They don't share context by default -- they hand off work through artifacts (GitHub issues, pull requests).

Persistent memory. Every entity (project) maintains a MEMORY.md file with curated architectural knowledge, daily session logs, and context files for decisions and gotchas. Agents read these on startup and write back what they learn. Knowledge accumulates across sessions.

State machine, not chat. Features move through a defined lifecycle (plan → design → build → review → ship → done) with approval gates and automatic phase transitions. The daemon tracks progress, spawns the right agent at the right time, and handles failures.

Multi-project isolation. Each project ("entity") gets its own configuration, Discord channels, memory, secrets vault, and Git repos. The daemon manages all of them simultaneously.

03 Architecture

The system is a Node.js daemon that manages agent sessions, routes Discord messages, tracks feature state, and exposes an HTTP API. Everything runs on a single machine.

The Daemon

A Node.js process managed by launchd (macOS service supervisor). On startup, it loads the global config and entity registry, initializes the session manager, task queue, feature manager, Discord bot, bot pool, Commander process, PR review cron, and HTTP server. The daemon handles graceful shutdown with drain mode: it waits for active agents to finish before killing tmux sessions, and persists pool state so bots can be resumed on restart.

Bot Pool

Ten Discord bots (pool-0 through pool-9), each with its own token and tmux session. When a feature enters an interactive phase (like build), the daemon assigns a free pool bot to a Discord work room channel. The bot runs a Claude Code session inside tmux, enabling real-time conversation between the user and the agent. Bots cycle through three states:

StateMeaning
freeAvailable for assignment
assignedRunning an active Claude Code session in a channel
parkedSession ended but assignment preserved for potential resume

The pool includes a health monitor that detects dead tmux sessions, and a nickname handler that sets each bot's Discord display name to match the active agent archetype (so users see "Builder" in the channel, not "lf-3").

Session Manager

Spawns Claude Code processes with the correct agent identity, model tier, DNA/skills, entity context, and session ID. Each session gets:

  • An --agent flag pointing to the archetype's soul file
  • Model and thinking flags (e.g., --model opus --think high)
  • An --append-system-prompt with entity context, memory paths, and daily log references
  • The prompt piped via stdin to avoid arg-length limits

Task Queue

A priority queue with configurable concurrency limits (default: 5 active sessions, 20 max queued). Tasks are sorted by priority (critical > high > medium > low), then by submission time. When a session completes, the queue processes the next pending task and emits a drain event so features blocked on queue capacity can retry.

State Persistence

Three JSON files in ~/.lobsterfarm/state/:

FileContents
features.jsonAll feature states (phase, assignments, PR numbers, session IDs)
pool-state.jsonBot assignments -- which bot is in which channel, for which entity

Additionally, each entity has a session-log.jsonl file -- an append-only log of every session spawned, with timing, archetype, phase, and exit code.

05 Feature Lifecycle

Every feature follows a state machine from idea to shipped code. The daemon manages transitions, spawns the right agent at each phase, and handles failures.

Phase Details

Plan phase

The Planner agent runs a socratic discovery session based on the GitHub issue. It outputs a detailed spec as an issue comment, including acceptance criteria, technical approach, and scope boundaries. This phase has an approval gate -- the user must approve the spec before the feature advances.

Transitions: plan → design (if UI-related labels present) or plan → build (skip design).

Design phase (optional)

The Designer creates coded prototypes using the entity's design system. This phase is optional -- it only runs if the feature has labels like ui, frontend, brand, or design. Also has an approval gate.

Transitions: design → build

Build phase

The Builder implements the feature in an isolated git worktree. A pool bot is assigned to a Discord work room for interactive collaboration. The Builder reads the spec from the GitHub issue, writes code and tests, and opens a PR. No approval gate -- the PR itself is the handoff artifact.

Entry actions: create git worktree, assign Discord work room, assign pool bot.

Transitions: build → review

Review phase

The Reviewer (always ephemeral, fresh eyes) audits the PR. It posts a structured review with an outcome: approved, changes requested, or pending. The daemon detects the outcome and routes accordingly.

Transitions: review → ship (approved) or review → build (changes requested, bounces back).

Ship and Done

Ship runs cleanup actions (worktree removal, pool bot release, Discord topic reset) and auto-advances to Done. When a feature reaches Done, the daemon checks if any other features were blocked by dependency on this one and unblocks them.

Other mechanics

Feature dependencies. Features can declare depends_on relationships. A feature blocked by unresolved dependencies waits until all dependencies reach done, then auto-unblocks and spawns its agent.

Automated PR review. A cron job polls every 5 minutes for open PRs across all entity repos. When it finds an un-reviewed PR, it spawns a Reviewer session. The outcome (approved / changes requested) feeds back into the feature state machine.

Start phase override. Features can be created at any valid starting phase (plan, design, or build) to skip earlier phases when the spec is already written.

06 Memory & Context

Agents don't retain context between sessions by default. Memory is explicit: files on disk that agents read on startup and write back when they learn something.

Memory layers

LayerFilePurposeLifespan
Long-term MEMORY.md Curated architectural knowledge, patterns, key decisions Permanent (manually curated)
Context context/decisions.md Significant architectural and design decisions Permanent
Context context/gotchas.md Known issues, workarounds, things that bite you Permanent
Short-term daily/YYYY-MM-DD.md Session summaries, progress, decisions made today Rolling (agents read last 5)
Audit session-log.jsonl Every session spawned: timing, archetype, exit code Append-only

Auto-extraction

When a session completes, the daemon runs a lightweight Haiku call to extract learnings from the session and append them to the daily log. This happens asynchronously and doesn't block the feature lifecycle. The goal: even if an agent doesn't explicitly write to the daily log, a summary of what it accomplished gets captured.

Multi-entity isolation

Entities are fully isolated. Each has its own memory, config, channels, and secrets vault. Only the Commander agent has visibility across all entities. When an agent session is spawned, it receives entity-specific context via --append-system-prompt and --add-dir, ensuring it can only access its own entity's memory directory.

08 Extending LobsterFarm

The system is designed to be extended through files, not code changes. New agents, DNA, skills, and blueprints are all declarative.

Custom agent archetypes

Agent soul files live in ~/.claude/agents/. Each is a Markdown file that defines the agent's personality, role, and operating principles. To add a new archetype, create a new file and reference it in the config:

~/.claude/agents/auditor.md# Auditor -- Security Specialist

You are a security auditor. Your job is to review code
for vulnerabilities, check for secrets exposure, validate
authentication flows, and ensure data handling compliance.

You approach every review assuming the code has a vulnerability.
Your job is to find it.

DNA and skills

DNA files in ~/.claude/skills/ are auto-loaded based on task matching or explicitly invoked. They carry domain knowledge, coding standards, and workflow instructions. To add custom DNA:

  1. Create a skill definition file in ~/.claude/skills/
  2. Define trigger conditions (keywords, file patterns, explicit invocation)
  3. Reference it in the relevant phase config or agent definition

Blueprints

Blueprints define entity templates. The default software blueprint includes channel structure, archetype assignments, SOPs, and guidelines. Custom blueprints can define different team compositions for different project types (e.g., a data-pipeline blueprint might emphasize the Operator archetype over the Designer).

Hook system

Claude Code's PreToolUse and PostToolUse hooks add safety guardrails. The default configuration includes hooks that:

  • Prevent committing directly to main (enforces worktree workflow)
  • Block creation of new branches in the main repo directory
  • Validate tool usage patterns

HTTP API

The daemon exposes a REST API on localhost:7749 for programmatic control. Any tool that can make HTTP requests can create features, check status, approve phases, or manage the bot pool. This enables integration with external systems, custom dashboards, or CI/CD pipelines.

09 Limitations & Roadmap

An honest look at where the platform is and where it's heading.

Current limitations

Single machine. The daemon, all tmux sessions, and all agent processes run on one macOS machine. There's no distributed mode, no remote workers, no cloud deployment. The machine's resources (CPU, memory, API rate limits) are the ceiling.

macOS-focused. The daemon relies on launchd for process supervision and tmux for TTY multiplexing. The lf init wizard handles macOS-specific concerns like Full Disk Access via TCC. Linux support is theoretically possible but untested and would require systemd integration.

Requires Claude Code subscription. Every agent session is a Claude Code process. The platform doesn't work with other AI providers or self-hosted models. Claude Code's Max plan is effectively the operating cost.

Discord as primary UI. The user interface is Discord channels. There's no web dashboard, no CLI for real-time status beyond curl localhost:7749/status. Discord's rate limits and message formatting constraints apply.

Nascent memory system. The memory architecture (MEMORY.md, daily logs, context files) works but requires manual curation to stay useful. There's no semantic search, no automatic pruning of stale knowledge, and no cross-entity knowledge sharing beyond what the Commander manually transfers.

Approval gates are manual. Phase transitions that require approval (plan and design) wait for a human to explicitly call the approve endpoint. There's no notification timeout or escalation if approval is delayed.

What's working well

Feature lifecycle. The state machine reliably drives features from idea to merged PR. The review bounce (changes requested sends it back to build) handles the most common iteration pattern.

Bot pool. The 10-bot pool with tmux sessions provides genuine interactivity. Agents can have real conversations, show progress, ask questions. The assign/park/resume lifecycle handles daemon restarts gracefully.

Specialization. Giving each agent a focused role with appropriate DNA produces noticeably better output than a single generalist agent. The Reviewer in particular benefits from having no prior context -- it catches things the Builder's tunnel vision misses.

Where it's heading

  • Web dashboard -- real-time view of features, agents, pool status, and session logs
  • Smarter memory -- semantic search over entity knowledge, automatic relevance ranking for context injection
  • Multi-machine support -- distributed bot pool across multiple machines
  • More phase types -- testing phases, staging deployment, user acceptance
  • Better observability -- token usage tracking, cost attribution per feature, performance metrics
